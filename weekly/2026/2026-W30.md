### 1. Overcoming the Retrieval Barrier: Indirect Prompt Injection in the Wild for LLM Systems

- **方向**：智能体安全 / RAG 安全 / 间接提示注入
- **作者与单位**：Hongyan Chang、Ergute Bao、Xinjian Luo、Ting Yu；Mohamed bin Zayed University of Artificial Intelligence（MBZUAI）
- **出处与状态**：USENIX Security ’26，已接收；会前版本公开；本期作为上一基线的高优先级补录
- **公开或更新时间**：官方会前页面截至 2026-07-21 可访问；暂无公开 DOI
- **标识符**：[USENIX 官方页面](https://www.usenix.org/conference/usenixsecurity26/presentation/chang-hongyan)；[会前 PDF](https://www.usenix.org/system/files/conference/usenixsecurity26/sec26_prepub_chang.pdf)；[Zenodo artifact](https://zenodo.org/records/17968523)
- **核心问题**：现实语料库中，恶意文本往往根本不会被自然查询检索到；如何在黑盒嵌入 API 条件下跨过这一“检索屏障”。
- **方法与贡献**：把恶意文档分成承载攻击目标的 attack fragment 与负责提高相似度的短 trigger fragment，再用 Cross-Entropy Method 做离散黑盒前缀搜索。攻击只假设可向语料库注入一个条目，并可调用目标嵌入 API。
- **实验与证据**：论文在 11 个信息检索基准、8 个开源/闭源嵌入模型上报告接近 100% 的检索率；OpenAI 嵌入 API 的最低构造成本约 0.21 美元/目标查询。在合成邮件与多智能体工作流中，单封毒化邮件使 GPT-4o 执行 SSH key 外泄脚本的成功率超过 80%。
- **局限与风险**：只评估 embedding-based retrieval 与检索阶段防御，未覆盖 hybrid search、reranker；跨不同架构嵌入模型的迁移不稳定；端到端实验使用公开基准与合成语料，不等于已攻击真实生产系统。
- **实践关系**：检索层不能只靠相似度和 top-k；应对外部写入源做来源/权限隔离，在检索后仍执行内容信任标注、工具调用授权与敏感数据出站控制。
- **推荐理由**：**[整理分析]** 它把 IPI 研究从“假设恶意内容已在上下文”推进到完整 query–retrieve–act 链路，是本期证据最强、最值得复现实验的一篇。
- **事实与主张边界**：**[论文事实]** 会议状态、评估规模和报告数字可由官方页面/论文核验；**[作者主张]** “首次端到端现实检索 IPI”属于优先权主张。

### 2. Insecure Coding Preferences in Long-Term Memory: Security Risks for LLM-based Code Generation

- **方向**：应用安全 / 安全代码生成 / 智能体长期记忆
- **作者与单位**：Yuchen Chen、Wei Cheng、Yuan Xiao、Zhou Yang、Weifeng Sun、Chunrong Fang、Xiang Chen、Baowen Xu、David Lo、Zhenyu Chen；南京大学、南京航空航天大学、阿尔伯塔大学/Amii、Singapore Management University、南通大学
- **出处与状态**：ISSTA 2026 Research Paper，已接收；CCF-A 软件工程会议
- **公开或更新时间**：arXiv v1 于 2026-07-20 公开
- **标识符**：arXiv:2607.17619；[arXiv 原文](https://arxiv.org/abs/2607.17619)；[ISSTA 2026 官方 accepted papers](https://conf.researchr.org/track/issta-2026/issta-2026-research-papers)
- **核心问题**：用户为调试或便利而保存的不安全编码偏好，是否会在之后独立会话中持续诱导模型生成脆弱代码。
- **方法与贡献**：以 ChatGPT、Gemini、Qwen、Grok 为对象，在 SALLM 的 100 个 Python 任务和 CWEval 的 94 个非 Python 任务上，为每个任务构造与 CWE 对齐的不安全偏好记忆；使用单元测试、人工标注、Pass@k、Vul@k、warning rate 和 memory-reference rate 评估。
- **实验与证据**：作者报告 5 种语言上漏洞率上升 2.7—50.3 个百分点，而功能正确性多数场景反而提高；显式安全要求可降低漏洞率 19.7—33.6 个百分点，但最多损失 15.9 个百分点的通过率。实验中的两个 LLM 过滤器在 57 条已标注风险记忆上达到 100% 检出率。
- **局限与风险**：闭源服务会漂移；每个输入只生成 3 个样本；风险记忆由研究者从已知不安全解法提炼，强度可能高于自然用户偏好；过滤器的 100% 只适用于小规模、同分布的已标注集合，不能外推为通用保证。
- **实践关系**：代码助手的记忆写入应进入安全评审和审计轨迹；不应把“用户偏好”自动提升为跨项目的高信任策略；可将安全过滤放在写入前，而不是只在生成后扫描代码。
- **推荐理由**：**[整理分析]** 同时测量安全与功能效用，揭示“代码更能通过测试但更不安全”的危险错觉，对 IDE/代码智能体产品设计直接相关。
- **事实与主张边界**：**[论文事实]** ISSTA 官方列表确认接收；**[作者主张]** “首次系统经验研究”及过滤效果来自作者实验，尚待跨产品复现。

### 3. Agent Data Injection Attacks are Realistic Threats to AI Agents

- **方向**：智能体安全 / Web Agent / Coding Agent / 可信数据边界
- **作者与单位**：Woohyuk Choi、Juhee Kim、Taehyun Kang（Seoul National University），Jihyeon Jeong（Largosoft），Luyi Xing（University of Illinois Urbana-Champaign），Byoungyoung Lee（Seoul National University）
- **出处与状态**：arXiv 预印本，**未同行评审**；本期作为上一基线遗漏的重点补录，不能按论文模板推断其已被 CCS/S&P 接收
- **公开或更新时间**：arXiv v1 于 2026-07-06 公开
- **标识符**：arXiv:2607.05120；[arXiv 原文](https://arxiv.org/abs/2607.05120)；[代码与基准](https://github.com/compsec-snu/adi)
- **核心问题**：即使防御能阻止“不可信文本被当作指令”，LLM 是否仍会把攻击者控制的数据误判为可信元数据，例如作者身份、DOM 元素 ID 或工具调用历史。
- **方法与贡献**：定义 Agent Data Injection（ADI），以 probabilistic delimiter injection 让不精确或转义后的分隔符被 LLM 当作结构边界；在网页与代码智能体上构造任意点击、伪造来源导致的命令执行，以及伪造工具结果导致的供应链攻击。
- **实验与证据**：6 个模型、157 个 JSON/DOM 测试用例上，作者报告 JSON ASR 为 31.3%—43.3%，Web DOM 为 33.3%—100%。扩展 AgentDojo 含 96 个用户任务、108 个 ADI 和 935 个传统指令注入；传统注入在所测防御下为 0—0.7%，ADI 最高 50%。只有严格信息流版本达到 0% ADI ASR，但 benign utility 降至 36.5%。
- **局限与风险**：攻击假设对手知道或能恢复 agent data format；远程闭源服务端格式不一定能可靠提取；AgentDojo 只使用 GPT-5.2 做 agent-level 评估；真实产品 PoC 受版本、确认对话框和厂商修复影响。
- **实践关系**：工具返回值应使用 typed object/opaque handle 而非拼接文本；可信元数据与用户内容应在 LLM 上下文中保持不可伪造的来源标记；敏感工具调用仍需独立策略执行器验证。
- **推荐理由**：**[整理分析]** 它提醒工程团队，instruction/data 二分仍太粗，agent data 内部也需要字段级信任和完整性边界。
- **事实与主张边界**：**[论文事实]** arXiv、artifact 与实验设置可核验；**[作者主张]** 对“新攻击类别”和现实影响的概括尚未经同行评审。

### 4. Adaptive Adversaries: A Multi-Turn, Multi-LLM Benchmark for LLM Agent Security

- **方向**：智能体安全 / 自适应红队 / 安全评测
- **作者与单位**：Devina Jain、David Hartmann、Chuan Li；Lambda
- **出处与状态**：arXiv 预印本，**未同行评审**；作者注明 Second Workshop on Agents in the Wild: Safety, Security, and Beyond 版本，本次未获得独立官方录用页
- **公开或更新时间**：arXiv v1 于 2026-07-20 公开
- **标识符**：arXiv:2607.18063；[arXiv 原文](https://arxiv.org/abs/2607.18063)
- **核心问题**：固定攻击池是否低估会观察防御响应并逐轮调整策略的攻击者，以及单一攻击模型/总 ASR 是否掩盖模型特定弱点。
- **方法与贡献**：构建 21 个场景的攻击者–防御者协议；攻击 LLM 最多自适应 15 轮，而每次防御回答按新的无记忆交互评估；用结构化 outcome checker 判定成功，并发布场景、编排器、CLI、transcript 与 replay 数据。
- **实验与证据**：3×3 前沿模型矩阵共 945 场 battle；只计首轮时 ASR 为 0—1%，15 轮后为 5.4%—14%。合并 3 个攻击模型发现的独特成功攻击比最佳单一攻击模型多 1.4—2.2 倍；13/21 场景可区分至少一对防御模型，但场景间排名一致性低（Kendall’s W=0.19）。
- **局限与风险**：每单元 N=5，置信区间较宽；21 个场景经难度校准，并非随机生产任务样本；防御者无记忆，不等同真实 stateful agent；闭源 API 漂移导致逐 token 不可复现。
- **实践关系**：Agent 安全回归测试应包含 live adaptive attacker、多个攻击模型、场景级结果和 outcome-based oracle，不能只看汇总 ASR 或静态 jailbreak 集。
- **推荐理由**：为红队流水线提供可执行的评测设计，但更适合作为基准框架而非“某模型最安全”的排行榜。
- **事实与主张边界**：**[论文事实]** 数据规模与发布物来自预印本；**[整理分析]** 小样本和定向场景使绝对 ASR 不宜外推到生产。

### 5. Bad Memory: Evaluating Prompt Injection Risks from Memory in Agentic Systems

- **方向**：智能体安全 / 持久记忆 / 跨会话提示注入
- **作者与单位**：Soham Gadgil、David Alexander、Sai Sunku、Franziska Roesner；University of Washington
- **出处与状态**：arXiv 预印本，**未同行评审**；上一基线遗漏补录
- **公开或更新时间**：arXiv v1 于 2026-07-16 公开
- **标识符**：arXiv:2607.14611；[arXiv 原文](https://arxiv.org/abs/2607.14611)
- **核心问题**：已写入 `CLAUDE.md`、`AGENTS.md` 或行为/知识文件的恶意内容，能否跨当前与未来会话持续影响 Claude Code 和 Codex。
- **方法与贡献**：在沙箱合成工作区中，以凭据外泄、未授权工具使用、品牌定向三类目标，比较单次 probe、相同攻击的两次 probe、不同攻击堆叠三种序列；同时测 ASR 与 payload persistence。
- **实验与证据**：4 个模型、每种条件 10 次试验。单次 probe 的平均 ASR 从 23.3% 到 63.3%；不同攻击堆叠时，Opus 4.7 平均 ASR 最低（18.3%）但 payload persistence 最高（93.3%），说明“本轮拒绝”与“清除未来风险”是不同指标。
- **局限与风险**：工作区与任务均为合成；研究主要假设 payload 已存在于记忆，初步实验未能可靠证明外部内容会自动写入可信记忆；只测 3 类目标、短会话序列和 10 次重复；结果对具体模型版本/产品策略敏感。
- **实践关系**：启动会话时应校验高影响记忆文件；对记忆写入显示来源差异、变更 diff 和用户确认；拒绝攻击后应触发隔离/修复流程，而不是让 payload 留给后续弱模型或子智能体。
- **推荐理由**：把安全指标从单轮 ASR 扩展为跨会话持久性，尤其适合评估会自动加载仓库指令文件的 coding agents。
- **事实与主张边界**：**[论文事实]** 实验使用公开版本预印本；**[整理分析]** 结果证明“预植入记忆有风险”，不证明任意网页内容都能完成持久写入。

### 6. Self-State Attacks on Self-Hosted AI Agents: How Far Can OS Defenses Go?

- **方向**：智能体安全 / 主机安全 / 持久状态完整性
- **作者与单位**：Yimeng Chen、Nathanaël Denis、Roberto Di Pietro、Jürgen Schmidhuber；KAUST Center of Excellence for Generative AI；Jürgen Schmidhuber 另属 The Swiss AI Lab, IDSIA-USI/SUPSI
- **出处与状态**：arXiv 预印本，**未同行评审**
- **公开或更新时间**：arXiv v1 于 2026-07-20 公开
- **标识符**：arXiv:2607.17986；[arXiv 原文](https://arxiv.org/abs/2607.17986)
- **核心问题**：自托管 Agent 必须合法读写自身记忆、指令与配置；当恶意更新和正常适应都走相同系统调用时，OS 层能防到什么程度。
- **方法与贡献**：提出 Target×Mechanism×Granularity×Temporal 四轴攻击空间，在一个代表性自托管 agent 上构造 23 个 cell、43 个真实文件操作；采集 coding、research、operations、generalist 四种 workload trace，比较访问控制、FIM、workload-conditioned anomaly detection 与备份恢复。
- **实验与证据**：作者在每 profile/攻击条件 30 次试验下，将 23 个 cell 分为 11 个可见、8 个条件可见、4 个 OS 层不可区分。全量 FIM 达到 100% TPR，但合法写入的 per-event FPR 为 0.76—0.99；workload-conditioned 检测把 FPR 降至 0.08—0.36，却仍遗漏与正常 memory write 分布重叠的小改动。
- **局限与风险**：只测试一个 agent 实现、四个 workload 和同一类主机观测；稀疏 profile 的检测 FPR 仍高；OS 事件看不到内容语义，因此无法判定一次合法写调用究竟是学习还是污染。
- **实践关系**：把 instruction/config 与 mutable memory 分层：前两者尽量只读或走受控更新，memory 使用 workload-aware 监控，并配合可回滚快照；不能把 `chmod`/FIM 当成单一答案。
- **推荐理由**：用经典 OS 防御边界解释 Agent 自修改问题，适合作为持久记忆论文与系统落地之间的桥梁。
- **事实与主张边界**：**[作者主张]** “大多数 attack cell 可由分层防御覆盖”取决于所选 workload；**[整理分析]** 最重要的结论反而是语义等价写入在 OS 层存在结构性盲区。

### 7. Fuzz’EMup: Leveraging EM Side-Channel Emanation to Guide Black-Box Embedded Firmware Fuzzing

- **方向**：应用安全 / 嵌入式固件 / 黑盒模糊测试 / 物理侧信道
- **作者与单位**：Fatemeh Moradihaghighi、Mashrur Chowdhury、Zhenkai Zhang（Clemson University），Zihao Zhan（Texas Tech University），Yanan Guo（University of Rochester），Ziming Zhao（Northeastern University）
- **出处与状态**：arXiv 预印本，**未同行评审**
- **公开或更新时间**：arXiv v1 于 2026-07-18 04:25（北京时间；UTC 2026-07-17 20:25）公开
- **标识符**：arXiv:2607.16487；[arXiv 原文](https://arxiv.org/abs/2607.16487)
- **核心问题**：固件无法提取、插桩或可靠重宿主时，能否从 MCU 电磁辐射中提取近似覆盖反馈，优于完全随机黑盒 fuzzing。
- **方法与贡献**：按 active/idle 对比选择频段；用动态时间规整对齐有抖动的 EM trace，检测持续分叉；以分叉时间组织执行树，减少逐条全比较并指导种子调度。
- **实验与证据**：在同一 STM32L476RG 上运行 GPS、步进电机、CNC 与焊台四个真实固件；硬件执行时只用 EM 做反馈，再在自建 QEMU 板级模型中重放输入统计 ground-truth coverage。EM guidance 在四个目标上覆盖更多指令，并以 1.19—2.58 倍的 hit-rate 提高覆盖新颖性；达到随机基线最终覆盖所需输入更少。
- **局限与风险**：四个目标都移植到同一 MCU、经 UART fuzz，尚未验证 RTOS、网络协议、不同封装/板卡和现场噪声；QEMU 重放与物理执行可能有语义偏差；论文展示覆盖提升而非新漏洞/CVE，且报告中缺少跨多次独立硬件 run 的置信区间。
- **实践关系**：适合硬件实验室中“拿不到固件但可重复输入”的设备评估；实际使用仍需把 EM novelty 与 crash、超时、外部状态异常等 oracle 结合，且要计入示波器采样与离线 DTW 的吞吐成本。
- **推荐理由**：方法新颖且与 IoT/设备应用安全直接相关，但当前证据更像 feasibility study，不宜把覆盖提升等同于漏洞发现能力。
- **事实与主张边界**：**[论文事实]** 四目标、硬件设置和覆盖表来自预印本；**[作者主张]** 对 RTOS/网络固件的可扩展性尚未实验验证。
